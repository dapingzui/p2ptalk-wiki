# Script表达能力边界

## 核心结论

Bitcoin Script 的重点不是“像不像通用计算机”，而是：
- 能否稳定停机
- 能否被快速验证
- 能否作为共识系统里的约束条件执行

所以它首先是 **验证语言 / 谓词系统**，而不是为了追求最大表达能力而设计的计算模型。

## 为什么“阿克曼函数”不能直接推出图灵完备

文博批判了一个常见跳步：

> "能写出阿克曼函数，不等于它就是图灵完备。"
> — **文博**

这里至少要分清四层概念：

1. **原始递归**
   - 一类受限但结构清晰的可计算过程
2. **超出原始递归**
   - 只能说明表达力比更弱系统强
3. **图灵可计算 / 全可计算**
   - 涉及一般意义上的无界可计算过程
4. **图灵完备**
   - 系统自身可以模拟通用计算

因此：
- “能表达增长极快的函数” ≠ “能模拟任意程序”
- “某个例子看起来很复杂” ≠ “系统具备无界控制流与无界存储”

Ackermann function 最多只能被用来说明某种表达能力，不足以单独证明 Bitcoin Script 图灵完备。

## 对“假本聪 / 全图灵机”论证的批判

文博批判的核心，是把外部能力偷记到协议本体上：

> "把外部展开、外部随机源、外部无限纸带，算成 Script 自己的能力，这是偷换。"
> — **文博**

典型偷换包括：
- **外部展开**：由外部程序把很多步骤预先展开，再喂给 Script
- **外部状态机**：把记忆和状态转移放在链外系统里
- **外部随机源**：把随机性放在外面，再把结果塞进脚本
- **外部无限纸带**：把不受限存储能力放到脚本外部

这些都不能算作 Bitcoin Script 的**原生能力**。

## 为什么矿工不需要一个“更强的计算机”

从矿工/节点视角，真正重要的是：
- 可停机
- 可预测资源开销
- 易验证
- 不把共识层变成任意程序执行器

所以脚本越接近“验证某个条件是否成立”，越符合共识系统需求。

这也是为什么“脚本是谓词”比“脚本是不是图灵完备”更抓住本质。

## 对 Atom / P2PTalk 的启发

如果 Atom 以后引入复杂业务逻辑，需要先区分三层：

1. **协议原生验证层**
   - 哪些条件是系统自身可验证的
2. **链外辅助层**
   - 哪些步骤只是外部工具帮忙展开
3. **业务编排层**
   - 哪些复杂性来自外围 agent / workflow，而不是共识本体

如果不分清，就会把外围系统的能力误判成协议本体能力。
