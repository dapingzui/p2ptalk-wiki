什么叫做等式，也就是左边和右边相等，又如何定义相等。例如 P2PKH 脚本，

input script:
	PUSHDATA(Sig) // 将签名压入栈
	PUSHDATA(PubKey) // 将公钥压入栈
output script: // 交易的来源
	DUP  // 把栈顶的元素（Pubkey）复制一遍
	HASH160 // 把栈顶元素弹出并取哈希并压入栈
	PUSHDATA(PubkKeyHash) // 将公钥哈希值（来源输出脚本收款人的哈希，也就是输入脚本的付款人）
												// 压入栈
	EQUALVERIFY // 弹出栈顶两个元素 比较是否相等
	CHECKSIG // 弹出栈顶两个元素 用公钥检查签名是否正确 如果正确则返回 true

按照上面的理论是否就是是 output script = input
script，首先我都不讲等式实际上是一个定义或方程，最终不能产生布尔值，这里能产生布尔值的只有：

   - HASH160(pubkey) == pubKeyHash
   - CHECKSIG（sig, pubkey）== true

这里的校验发生在脚本内部执行进行的元素比较，而不是 scriptPubKey =
scriptSig，这种“脚本对脚本的等式”，不理解这里如何能称之为等式而布尔值函数。

实际上是

$$ \mathrm{ValidSpend}=P_{\mathrm{out}}(\mathrm{input
},\mathrm{tx\_context})\in\{\mathrm{true},\mathrm{false}\} $$

这样的布尔值函数形式，锁定脚本作为布尔值函数，而 input 和 交易上下文作为其参数，最终得到一个布尔值来决定其是否能花费。

什么叫做等式，用等号连接的表达式。什么是等号，等号就是一个符号，就是一个操作符。
在C++里面有一个功能是覆写操作符，也就是同样的符号，在我这个场景下我可以给它一些额外的操作或功能。

基于此去理解比特币的谓词逻辑，它的锁定脚本和解锁脚本就是一个等式，一个将锁定脚本和解锁脚本连接起来经过操作符的处理逻辑最终的到一个true或false功能的一个表达式。

这里为什么用这个视角去看呢？首先锁定脚本和解锁脚本在比特币上面出现的时间是不一样的，一定是锁定脚本先上链，然后有解锁脚本和它"相等"才能解锁一笔UTXO，而相等这个功能，它不是锁定脚本定义的，也不是解锁脚本定义的，它就是比特币自身去定义的。相当于比特币定义了操作符，锁定脚本提供了等式的左侧，需要提供等式右侧的人提供合适的解锁脚本才能让等式成立。如果用函数的视角来看，那么锁定脚本和解锁脚本同时作为参数应该是同时出现的。

这里很讨厌教条主义，因为对任何问题的抽象和创新解释一定是和之前历史的解释不同的，一味拿历史的解释去对照的话一定能挑出毛病的。

还有一句不太理解  “锁定脚本作为布尔值函数，而 input 和 交易上下文作为其参数，最终得到一个布尔值来决定其是否能花费。”

这里我不认为锁定脚本是那个函数，而且我不理解交易上下文是什么，因为在我的理解里面UTXO是存粹的，没有被环境中的上下文所污染的一个结构，所以我不理解这个上下文是什么。

你说比特币重定义了等号操作符，但我要直接指出：
*比特币从来没有用等号将锁定脚本和解锁脚本这两部分连接起来，也没有通过覆写等号操作符来实现脚本的拼接和执行。*

不是等号被重定义了，而是等号根本没有在这两部分之间扮演过任何角色。没有任何一个操作码、没有任何一个协议层的机制，是把锁定脚本和解锁脚本作为整体用等号连接起来的。你说的"操作符重定义"是你自己加进去的，比特币的实现里找不到对应物。

这里的问题不是创新解释和教条主义之争，而是*你的论证顺序反了*：

你先断定"这是一个等式"，然后发现等式需要一个等号，再去设想比特币重定义了这个等号来自圆其说。这是在用结论推导前提，而不是从事实推导结论。

如果你是从执行过程出发去观察，你会看到的是：两段脚本拼接，压栈，顺序执行，看栈顶。这个过程里没有任何一步在"比较左边和右边是否相等"。等式这个框架不是被比特币用新方式实现了，而是从一开始就没有出现过。
