同意叶鹏观点，用图论来表示拥有atom的用户的联系是不够的，在一个空间里面计算向量，矩阵看起来确实是一个解，增加一个atom等于增加一个向量，向量在空间里面的变化会导致形状改变，矩阵能在3维完成这个工作，这里需要讨论的是，atom是不是无限维，是否可以压缩，是否可以计算。听懂的可以拉会讨论。

我觉得ATOM跟比特币有本质上的区别。所以，atom本身不存在业务需求，所以atom本身不应该进入mempool而是需要atom支持的交易进入mempool，作为业务的开始。

>>>> 哈希算法简略过程：将输入按 64B 分块；在消息末尾做 padding（加 1 bit、补 0、再加 64-bit 长度）使总长度为 64B 
>>>> 的整数倍。初始化链值 H 为 8 个固定 32-bit 常量（IV）。对每个 chunk：先把 chunk 解析为 16 个 32-bit 字 
>>>> W[0..15]，再扩展得到 W[0..63]；把 H 拷贝到工作寄存器 a..h；进行 64 轮迭代，第 t 轮使用 W[t] 和常量 K[t] 
>>>> 更新寄存器；轮结束后将 a..h 分别加回到 H 得到新的 H。所有 chunk 处理完后，输出最终 H 作为摘要。
>>>>
>>>>    - 原始挖矿：固定区块模板（含 merkle root / time / nBits），内循环遍历 nonce；nonce 
>>>>    空间扫完通常改coinbase 内的extranonce  → 改 merkle root → 重复。 
>>>>    - AsicBoost：构造一组 work items，使它们 *chunk2 的 Message 相同*但 *midstate 不同*。对每个 
>>>>    nonce：chunk2 的 schedule（Expander1）算一次，然后对 n 个 midstate 各跑一次 Compressor1 + 
>>>>    第二遍 SHA，从而把 Expander1 的成本摊薄到 1/n。 
>>>>    - SegWit 的影响：SegWit 区块要求 coinbase 包含 witness 
>>>>    commitment，使“通过交易重排来快速刷 merkle root 候选”的 covert 路线变得不实用/与打包 segwit 交易存在冲突动机。 
>>>>    - nVersion 的影响：如果有一段 nVersion 位可用于 version rolling 
>>>>    且不被当作软分叉信号（BIP320 的设想），则可以走 *overt AsicBoost*：用滚 nVersion 来产生多组 
>>>>    midstate，*不需要* merkle root 碰撞，因此可以在打包 segwit 交易的情况下继续获得 AsicBoost 
>>>>    的复用收益。 
>>>>
>>>> 看完之后的想法：这个算法是利用了哈希函数的分块，而默克尔根正好被分成两半，后一段只有四位，让修改交易撞出后四位相同的默克尔根成为可能性，如果分开后后半段比如是16

atom机制可以在链上进行存证。通过问题1的计算，atom的创建在链上是有一笔交易的，这里设计的atom是有存续时间的，在存续时间内所有的atom的交易都在mempool中，当atom的存续时间终止时，发行方需要整理所有的atom交易并将其做成一个默克尔树，为每一笔atom交易提供一个默克尔路径，然后将默克尔树的根hash和atom的一些信息组合成一笔比特币转账上比特币的链，这样atom的信息就在链上进行存证了，过段时间矿工就可以在mempool上将这个atom的相关交易从mempool中删除来释放本地的内存空间了。

*比特币最初的代码设计中，并不是只有mempool是唯一业务来源，实际存在多条并行的业务来源。所以形式化的问题：不应该是atom在mempool怎么实现全局状态。而是*
*在无中心、无强制全局共识的条件下，**atom 是否能产生“足够一致”的局部信任，以支持实际决策/业务？*


原因如下：

从代码结构与依赖关系看，mempool 与 atom 位于不同的功能层级：

mempool 围绕 CTransaction 实现节点本地的交易缓存与转发机制；

atom 则围绕 CUser / CReview 等非交易对象，实现链下的业务与声誉逻辑。

二者在对象类型、状态空间以及调用路径上彼此独立，不存在直接依赖或调用关系。


进一步看，mempool 并不是比特币系统中唯一的业务来源。

除了通过交易中继（mempool）进入系统的交易输入流外，代码中还设计了基于 pub/sub 的业务消息输入流，用于承载非交易类业务对象（如
product、review、atom 等market的业务）。


在代码实现上，这两类业务来源仅在最底层共享同一套 P2P 网络通信机制（消息封装、连接管理与转发），但在消息类型、处理路径与业务语义上完全分离。

因此，在早期设计里至少看到了两类业务来源：一类来源于交易流，另一类来源于非交易的业务消息流。

在这一框架下，atom 明确属于后者，其生成与传播独立于 mempool，也不应被建模为交易或交易池状态的一部分。
