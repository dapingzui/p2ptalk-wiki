难度调整、符合难度，那什么叫符合难度？代码中怎么体现的
1.对于btc中难度调整相关的内容，由亚鹏通过代码与讲解，有代码提供证据，btc网络中，假设说现在有我和亚鹏两个节点，亚鹏出了一个块，首先亚鹏得算出nounce组成区块，进行哈希，前面要保持多少个0，才能符合难度，然后2016个区块后进行难度调整。那怎么才叫符合难度呢，我需要对亚鹏计算出来的nounce进行“校验”，本来我理解这里是“校准”，但亚鹏觉得校验更准确，这里不纠结，也就是说在btc这种p2p网络中，每个节点对于其他节点出块都需要进行“校验”，之前注意力都放在了符合难度上了，默认节点的难度计算公式相同则节点之间都是符合难度的，其实这里每个独立的节点都需要“校验”，且一直保持的。

代码证据：0.1.0版本在接收到区块以后，会调用函数GetNextWorkRequired，这里会返回该区块的难度(nbit)字段应该是一个什么值，并在AcceptBlock中进行对比，
 if (nBits != GetNextWorkRequired(pindexPrev))
        return error("AcceptBlock() : incorrect proof of work");

如果对比不通过就直接报错。

会议中的“校验”？
2.对于会议来说，我们有没有进行“校验”，注意力有没有放到同一件事情上来，有，只有老史在做这件事，老史在“校验”所有人、节点，这里与上面说的btc是不一样的，我们节点与节点之间是没有进行过“校验”，比方说之前，我与亚鹏是很少开会的，但这两周，上周我与亚鹏沟通了market.cpp，这周沟通了btc难度调整代码，虽然说我们彼此在说彼此的事情，但是今天下午亚鹏发的邮件内容，“加强邮件回复形成反馈系统”有某种相似的地方，我强调节点之间需要进行“难度校验”，亚鹏强调“邮件反馈”，好像进行了某种难度校验？是不是幻觉我不知道，但我确实认为需要节点与节点之间需要去“校验难度”。

具体到会议中该怎么使用节点与节点之间的“难度校验”，能否形成会议规则，需要讨论
