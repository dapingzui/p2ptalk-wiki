你说比特币重定义了等号操作符，但我要直接指出： *比特币从来没有用等号将锁定脚本和解锁脚本这两部分连接起来，也没有通过覆写等号操作符来实现脚本的拼接和执行。* 不是等号被重定义了，而是等号根本没有在这两部分之间扮演过任何角色。没有任何一个操作码、没有任何一个协议层的机制，是把锁定脚本和解锁脚本作为整体用等号连接起来的。你说的"操作符重定义"是你自己加进去的，比特币的实现里找不到对应物。 这里的问题不是创新解释和教条主义之争，而是*你的论证顺序反了*： 你先断定"这是一个等式"，然后发现等式需要一个等号，再去设想比特币重定义了这个等号来自圆其说。这是在用结论推导前提，而不是从事实推导结论。 如果你是从执行过程出发去观察，你会看到的是：两段脚本拼接，压栈，顺序执行，看栈顶。这个过程里没有任何一步在"比较左边和右边是否相等"。等式这个框架不是被比特币用新方式实现了，而是从一开始就没有出现过。 ------------------------------ 回到谓词本身的定义。谓词在数学和计算机科学里的含义是明确的：*一个以某些变量为输入、返回布尔值的函数*。你代码里写的 isNull(x)、
isNumber(x) 就是谓词，没有任何歧义。中本聪说比特币脚本是谓词，意思也很直接：锁定脚本就是一个函数，解锁脚本提供的数据是它的输入，执行结果是 true 或 false。 这个定义从来没有要求谓词是一个等式，谓词就是布尔值函数，仅此而已。你说"比特币把谓词扩展了，谓词不再是布尔值函数而是一个等式"——但谓词扩展成等式，它就不再是谓词了。你不能一边说这是谓词，一边说谓词在这里是等式，因为等式本身不产生布尔值，你自己也承认了这一点。 ------------------------------ 另外我之前说的"锁定脚本作为布尔值函数，input 和交易上下文作为参数"，这个描述也有问题，需要收回来。 准确的描述应该是： ValidSpend=Execute(scriptSig∥scriptPubKey)∈{true,false} 锁定脚本不是独立的布尔值函数，解锁脚本也不只是它的参数。两者拼接后作为一个整体被执行，执行的结果才是布尔值。UTXO 本身是纯粹的，它只携带锁定脚本，不依赖任何外部上下文。把交易上下文拉进来作为参数，破坏了这个纯粹性，是我表述不当。 你真正观察到的是条件的发布和证明的提供在时间上分离，这个洞察是对的，但它撑不起等式这个词，也不需要去修改谓词的定义来容纳它。 胥亚朋 <xuyapeng110@gmail.com> 于2026年3月3日周二 14:28写道： > 什么叫做等式，用等号连接的表达式。什么是等号，等号就是一个符号，就是一个操作符。 > 在C++里面有一个功能是覆写操作符，也就是同样的符号，在我这个场景下我可以给它一些额外的操作或功能。 > > > 基于此去理解比特币的谓词逻辑，它的锁定脚本和解锁脚本就是一个等式，一个将锁定脚本和解锁脚本连接起来经过操作符的处理逻辑最终的到一个true或false功能的一个表达式。 > > > 这里为什么用这个视角去看呢？首先锁定脚本和解锁脚本在比特币上面出现的时间是不一样的，一定是锁定脚本先上链，然后有解锁脚本和它"相等"才能解锁一笔UTXO，而相等这个功能，它不是锁定脚本定义的，也不是解锁脚本定义的，它就是比特币自身去定义的。相当于比特币定义了操作符，锁定脚本提供了等式的左侧，需要提供等式右侧的人提供合适的解锁脚本才能让等式成立。如果用函数的视角来看，那么锁定脚本和解锁脚本同时作为参数应该是同时出现的。 > > 这里很讨厌教条主义，因为对任何问题的抽象和创新解释一定是和之前历史的解释不同的，一味拿历史的解释去对照的话一定能挑出毛病的。 > > 还有一句不太理解 “锁定脚本作为布尔值函数，而 input 和 交易上下文作为其参数，最终得到一个布尔值来决定其是否能花费。” > > > 这里我不认为锁定脚本是那个函数，而且我不理解交易上下文是什么，因为在我的理解里面UTXO是存粹的，没有被环境中的上下文所污染的一个结构，所以我不理解这个上下文是什么。 > > 陈默 <chenmoonmo@gmail.com> 于2026年3月2日周一 17:50写道： > >> 什么叫做等式，也就是左边和右边相等，又如何定义相等。例如 P2PKH 脚本， >> >> input script: >> PUSHDATA(Sig) // 将签名压入栈 >> PUSHDATA(PubKey) // 将公钥压入栈 >> output script: // 交易的来源 >> DUP // 把栈顶的元素（Pubkey）复制一遍 >> HASH160 // 把栈顶元素弹出并取哈希并压入栈 >> PUSHDAT

isNumber(x) 就是谓词，没有任何歧义。中本聪说比特币脚本是谓词，意思也很直接：锁定脚本就是一个函数，解锁脚本提供的数据是它的输入，执行结果是 true 或 false。 这个定义从来没有要求谓词是一个等式，谓词就是布尔值函数，仅此而已。你说"比特币把谓词扩展了，谓词不再是布尔值函数而是一个等式"——但谓词扩展成等式，它就不再是谓词了。你不能一边说这是谓词，一边说谓词在这里是等式，因为等式本身不产生布尔值，你自己也承认了这一点。 ------------------------------ 另外我之前说的"锁定脚本作为布尔值函数，input 和交易上下文作为参数"，这个描述也有问题，需要收回来。 准确的描述应该是： ValidSpend=Execute(scriptSig∥scriptPubKey)∈{true,false} 锁定脚本不是独立的布尔值函数，解锁脚本也不只是它的参数。两者拼接后作为一个整体被执行，执行的结果才是布尔值。UTXO 本身是纯粹的，它只携带锁定脚本，不依赖任何外部上下文。把交易上下文拉进来作为参数，破坏了这个纯粹性，是我表述不当。 你真正观察到的是条件的发布和证明的提供在时间上分离，这个洞察是对的，但它撑不起等式这个词，也不需要去修改谓词的定义来容纳它。 胥亚朋 <xuyapeng110@gmail.com> 于2026年3月3日周二 14:28写道： > 什么叫做等式，用等号连接的表达式。什么是等号，等号就是一个符号，就是一个操作符。 > 在C++里面有一个功能是覆写操作符，也就是同样的符号，在我这个场景下我可以给它一些额外的操作或功能。 > > > 基于此去理解比特币的谓词逻辑，它的锁定脚本和解锁脚本就是一个等式，一个将锁定脚本和解锁脚本连接起来经过操作符的处理逻辑最终的到一个true或false功能的一个表达式。 > > > 这里为什么用这个视角去看呢？首先锁定脚本和解锁脚本在比特币上面出现的时间是不一样的，一定是锁定脚本先上链，然后有解锁脚本和它"相等"才能解锁一笔UTXO，而相等这个功能，它不是锁定脚本定义的，也不是解锁脚本定义的，它就是比特币自身去定义的。相当于比特币定义了操作符，锁定脚本提供了等式的左侧，需要提供等式右侧的人提供合适的解锁脚本才能让等式成立。如果用函数的视角来看，那么锁定脚本和解锁脚本同时作为参数应该是同时出现的。 > > 这里很讨厌教条主义，因为对任何问题的抽象和创新解释一定是和之前历史的解释不同的，一味拿历史的解释去对照的话一定能挑出毛病的。 > > 还有一句不太理解 “锁定脚本作为布尔值函数，而 input 和 交易上下文作为其参数，最终得到一个布尔值来决定其是否能花费。” > > > 这里我不认为锁定脚本是那个函数，而且我不理解交易上下文是什么，因为在我的理解里面UTXO是存粹的，没有被环境中的上下文所污染的一个结构，所以我不理解这个上下文是什么。 > > 陈默 <chenmoonmo@gmail.com> 于2026年3月2日周一 17:50写道： > >> 什么叫做等式，也就是左边和右边相等，又如何定义相等。例如 P2PKH 脚本， >> >> input script: >> PUSHDATA(Sig) // 将签名压入栈 >> PUSHDATA(PubKey) // 将公钥压入栈 >> output script: // 交易的来源 >> DUP // 把栈顶的元素（Pubkey）复制一遍 >> HASH160 // 把栈顶元素弹出并取哈希并压入栈 >> PUSHDAT

脚本是谓词

中本聪的观点

中本聪说：比特币脚本是谓词。

文博的分析：

"比特币脚本是谓词，意思也很直接：锁定脚本就是一个函数，解锁脚本提供的数据是它的输入，执行结果是 true 或 false。"

谓词的形式化定义

锁定脚本（Locking Script）= 函数
解锁数据（Unlocking Data）= 输入
执行结果 = true 或 false

Bitcoin 脚本的谓词本质

脚本类型 作用 谓词解释
P2PKH 支付到公钥哈希 验证签名是否匹配公钥哈希
CHECKSIG 签名验证 验证签名是否有效
isNull(x) 空值检查 验证输入是否为空
isNumber(x) 数值检查 验证输入是否为数字

谓词不是等式

"这个定义从来没有要求谓词是一个等式，谓词就是布尔值函数，仅此而已。"

关键点： - 谓词 = 布尔值函数 - 返回 true 或 false - 等式不是谓词

对 Atom 系统的意义

Atom 系统中的"谓词"概念继承自 Bitcoin 脚本： - 锁定脚本 = 验证条件 - 执行结果 = TRUE/FALSE - 通过谓词验证实现信任

相关概念

  • atom/谓词
  • consensus/BitDNS与P2P市场
