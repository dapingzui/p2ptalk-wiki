中本聪的邮件里面有一句:The publish/subscribe "meet in the middle" mechanism was an interesting concept, but nothing remains that uses it. 这里有三个重点，第一个就是publish/subscribe，也就是发布订阅机制。第二个是meet in the middle。这个好像是分布式网络中才有的一个东西。第三个就是nothing remains that uses it，也就是没人用它了。 感觉这个地方值得研究一下。 首先就是publish/subscribe，订阅发布（Publish-Subscribe，简称Pub/Sub） 是一种消息传递模式，在这种模式中，消息的发送者（发布者）不直接将消息发送给特定的接收者（订阅者），而是将消息分类到不同的主题（topic）或频道（channel）。订阅者可以表达对一个或多个主题的兴趣，并接收相关消息，而不知道发布者的存在。 传统的pub/sub是异步的，非点对点的，一对多或者多对多的。这种消息传递模式有很多优点。 1. 松耦合，这个
特性很像我们通过邮件进行沟通，我不需要知道你在哪儿，是否在线，我都可以发送邮件，什么时候你们上线了就收到邮件了。 2. 可扩展。邮件的主题可以扩展，我们这里就是为新的主题新开一个邮件，参与人也随时可以添加或减少。 3. 灵活，支持一对多或多对多，支持消息过滤。 pub/sub的实现模式有2种，一种是推模式，也就是消息发布者主动广播消息，这种情况下实时性高，但是作为被动接收的订阅者需要实时在线，而且消息过多可能来不及处理。 第二种是拉模式，也就是订阅者去拉取消息，这种模式实时性差，但是订阅者掌握节奏，对于发布者来说负担更小，因为发布者一般会将消息放到一个中间件里面供订阅者拉取。但是订阅者通常会轮询消息，导致中间件负载过大。 一般实现会采用混合模式，即可以推，也可以拉。 胥亚朋 <xuyapeng110@gmail.com> 于2026年2月4日周三 16:18写道： > 在现有的比特币框架中嵌入一套ebay系统 > > 首先是商品信息，这里不修改比特币的任何结构，结合外部信息的话可以使用类似NFT的框架。 > > > 构造一笔特殊的转账，金额可以很小，转账的信息里面有一个URL，URL上面就是具体的商品信息，价格，以及商家联系方式相关。转账需要有一个locktime,这个locktime就是一个商品公示期,也就是这个商品在链上能存续的最小时间。 > > > 外部计算，需要这个业务的服务器可以自行下载链上交易，获取商品URL并从URL上面下载商品信息来对用户提供服务，保证信息源来源于比特币，附属信息可以由外部提供。并对外提供商品信息页面促进商品成交。 > > > 用户在商品信息页面选择商品，根据商品上面的商家联系方式联系商家并成交。如果成交，商家需要在locktime结束以后将这个比特币转账给用户的，代表用户购买了相关商品并标识这个商品已经售出，不能二次销售。至于商品成交的最终价格，购买订单这些详细信息不宜在链上留下痕迹，链上仅有历史商品的比特币，哪个地址购买了这个比特币这两个信息。 > > > 用户在获得商品以后，如果想要对商品进行评价，需要将带有商品信息的这个比特币销毁，并在销毁的转账里面添加自己的评价信息。商品不能被重复评价，已经被评价过的商品的比特币不能重复使用。如果用户没有对商品进行评价，该比特币可以重复使用，之前的商品买卖信息归为历史。 > > 商品的评价信息和商家的绑定需要外围服务去做，但是如果商品信息的URL的内容被修改就无法溯源，所以商品评价是有窗口期的，过了这个窗口期则评价无效。 > > 胥亚朋 <xuyapeng110@gmail.com> 于2026年2月3日周二 16:31写道： > >> 如何在比特币上面实现ebay那样的市场？ >> >> >> 这里把比特币作为一个互联互通的P2P网络来看，区块链就是比特币上面传递的相关消息，区块链可以当作是一种确定的消息，P2P网络上面还有一部分类似交易的信息，可以在网络上进行传递，但是可以不进块。 >> >> 角色分布应该有:商品发布者，商品购买者。 >> >> >> 商品发布者在网络上发布商品信息，用户可以在网络上查询获得商品信息，并与商品发布者联系购买商品，P2P网络不承担商品真实性相关检查。商品购买者可以对商品信息进行评价，最终评价会落实到商品发布者本人上面，这个应该就是atom的逻辑，给予商品发布者一定的信誉值。 >> >> 问题: >> 1. 商品信息是否上链？ 商品发布信息个人建议是不上链，但是如何让矿工们接收并传递商品信息是一个问题 >> 2. 成交信息是否上链？ 成交

发布订阅机制

Bitcoin 的发布订阅

Bitcoin 使用 Gossip Protocol（八卦协议）： - 无差别广播 - 所有节点收到所有消息 - 无订阅关系

Atom 的发布订阅

文博：

"关于商品 offerbook 的传播完全继承比特币的发布订阅和中间会面机制。"

Atom 的发布订阅特点： - 有订阅关系 - 通过中间会面转发 - 谓词验证过滤

发布订阅与匹配

发布订阅解决的问题： - 意图广播：我想要什么 - 能力广播：我能提供什么

匹配 = 意图 + 能力的对接

中间会是发布订阅的增强

中间会面 = 发布订阅 + 谓词验证 - 不是所有订阅者都收到 - 而是满足谓词才转发

  • broadcast/中间会面机制
  • market/匹配系统
